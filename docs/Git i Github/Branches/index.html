Branches (gałęzie) w Git

Co to jest flaga (flag), inaczej przełącznik (switch)?​

  • Parametry takie jak na przykład -c nazywamy flagami (flags) lub przełącznikami (switches).
  • Są one powiązane z komentadmi wydawanymi w konsoli w systemach Linux i Windows.
  • użycie flagi odpowiednio modyfikuje działanie danej instrukcji wydawanej w konsoli.

W kontekście Git (i ogólnie narzędzi CLI), mamy dwa rodzaje flag:

  • Flagi krótkie (short options): Są to pojedyncze znaki poprzedzone myślnikiem (-).

  • Przykłady:

    • -c (odpowiednik --create)
    • -m (odpowiednik --message przy commitach)
    • -b (odpowiednik --branch przy* checkout*)

  • Flagi długie (long options): Są to pełne nazwy opcji poprzedzone dwoma myślnikami (--).

  • Przykłady:

    • --create
    • --message
    • --branch

Co to jest branch (gałąź lokalna)?​

::: note Gałąź jako tor kolejowy

  • main/master = główny tor kolejowy (stabilna trasa).
  • branch = bocznica, gdzie możesz puścić pociąg testowy.
  • commit = kolejne stacje, przez które przejeżdża pociąg.
  • merge = zjazd z bocznicy z powrotem na główny tor. Dzięki gałęziom masz więc pewność, że eksperymentujesz na bocznicy, nie blokując głównej linii ruchu.

:::

Sprawdzenie gałęzi, na której jesteś​

  • git branch

Tworzenie nowej gałęzi​

  • git branch <nazwa>
  • Tworząc nazwę gałęzi stosuj kebab-case, czyli wszystkie słowa pisane są małymi literami, a łączone są znakiem -.

inny sposób:

  • git switch -c <nazwa>
  • tworzysz nową gałąź i przechodzisz do niej.

Przełączanie się na inne gałęzie​

  • git switch <nazwa>

Usuwanie gałęci​

  • git branch -d <nazwa> (bezpieczne usuwanie, jeżeli zmergowana, to zostanie usunięta)
  • git branch -D <nazwa> (delete force, czyli wymuszone usuwanie - gałąź zostanie usunięta, nawet jeżeli jest zmergowana)

Gałąź na Githubie (gałąź zdalna)​

Gałąź lokalna przesłana do repozytorium zdalnego:​

  • Jeżeli lokalnie zrobisz gałąź, wyślij ją na GitHuba:
  • git push origin <nazwa-nowej-gałęzi>

Wysyłanie (push) nowej gałęzi do zdalnego repozytorium (np. na GitHub):

  • nie musisz być na tej konkretnej gałęzi, którą chcesz wysłać,
  • możesz być na dowolnej gałęzi lokalnej i wciąż wysłać inną gałąź do zdalnego repozytorium.

git push origin <nazwa-galezi>

W tym poleceniu:

  • origin – odnosi się do zdalnego repozytorium, do którego wysyłasz gałąź (domyślnie origin).
  • nazwa-galezi – odnosi się do nazwy lokalnej gałęzi, którą chcesz wysłać do zdalnego repozytorium.

Gałąź zdalna "ściągnięta" do repozytorium lokalnego:​

  • twórz gałąź na Githubie,
  • utwórz na tej gałęzi nowy plik
    • git branch wyświetla wszystkie lokalne gałęzie
    • git branch -r wyświetla wszytkie zdalne gałęzie
    • git fetch aktualizuje informacje o zdalnych gałęziach w twoim lokalnym repozytorium (po tej instrukcji wylistuj repozytoria git branch -r)
    • git switch -c <nowa-gałaz> origin/nowa-galaz> - wyjaśnienie:
      • git switch -c <nowa-galaz> – tworzy nową lokalną gałąź o nazwie <nowa-galaz>.
      • origin/<nowa-galaz> – wskazuje na zdalną gałąź <nowa-galaz> z origin, która jest bazą do utworzenia lokalnej gałęzi.
      • np. git switch -c feature-login origin/feature-login
        • -c feature-login tworzysz nową lokalną gałąź o tej nazwie,
        • origin/feature-login mówisz Gitowi, że ma bazować na gałęzi zdalnej origin/feature-login.

nowsza syntaktyka uproszczona:

git switch --track origin/nazwa​

Git sam stworzy lokalną gałąź feature-login połączoną ze zdalną.

Gdy na lokalnej gałęzi zrobiłeś zmiany​

  • przełącz się na gałąź (lokalnie), która jest lokalnie i zdalnie
  • zmień zawartość jednego z plików (lokalnych)
  • będąc na tej gałęzi wykonaj
  • git push lub git push -u origin <nazwa-tej-galezi>
    • -u (--set-upstream) ustawia powiązanie (upstream) między lokalną gałęzią a zdalną
    • origin skrót do zdalnego adresu do repozytorium
    • <nazwa-tej-galezi> to nazwa gałezi, którą wysyłasz na GitHub

git merge​

  • git merge to polecenie, które pozwala na łączenie zmian z jednej gałęzi do innej.
  • Głównym celem git merge jest scalanie historii dwóch różnych gałęzi, tak aby wszystkie zmiany z obu gałęzi znalazły się w jednej wspólnej gałęzi.
  • Najczęściej git merge jest używany do:
    • Łączenia zmian z gałęzi funkcyjnych (feature branch) do głównej gałęzi.
    • Łączenia zmian z głównej gałęzi do innej gałęzi, aby zaktualizować ją o najnowsze zmiany.

Jak dochodzi do git merge​

Do git merge dochodzi, gdy:

  • chcesz połączyć zmiany z jednej gałęzi do innej.
  • masz przynajmniej dwie różne gałęzie z różnymi zmianami.
  • próbujesz zintegrować te zmiany, używasz git merge, aby utworzyć jeden spójny stan projektu.

Scenariusz 1 - bez konfliktu​

  • gałąź główna master
  • na gałęzi feature/new-function wprowadź zmiany w jakimś pliku
  • zmiany dodaj do poczekalni, a potem zrób commit
  • przełącz się na gałąź master (git switch master)
  • połącz gałęzie git merge feature/new-function
  • sprawdź wynik git log --oneline

Scenariusz 2 - konflikt w zmianach​

  • gałąź główna master w pliku np. A.txt ma zmiany, które są zacommitowane,
  • gałąź feature/galaz-z-konflitem w tym samym pliku A.txt ma inne zmiany, które zostały zacommitowane

Jeśli te same linie w tych samych plikach zostały zmienione w obu gałęziach, Git napotka konflikt.

Git wskaże pliki z konfliktami, a Ty będziesz musiał ręcznie wybrać, które zmiany zachować, a które usunąć.

Wyedytuj plik, który wygenerował błąd - przykładowy plik z konfliktem:

<<<<<<< HEAD
to pi e r w s z z y tekst TO JEST MASTE
=======
wwto jest tu to jest coś z FEATURE_NEW tekst
>>>>>>> feature/new-func
to jest nowy tekst
teraz trzy teraz jestem na githubie

z n a c z n i k i k o n f l i k t u

  • <<<<<<< HEAD
    • linia ta oznacza początek konfliktu
    • pokazuje, że poniższy fragment pochodzi z gałęzi, na której aktualnie się znajdujesz
  • =======
    • linia ta jest separatorem
    • oznacza koniec zmian z pierwszej gałęzi, a początek zmian z drugiej gałęzi, którą próbujesz scalić
  • >>>>>>> feature/new-func
    • linia ta oznacza koniec konfliktu
    • informuje, że powyższe linie po ======= pochodzą z gałęzi feature/new-func, czyli z gałęzi, którą próbujesz scalić.

Po ręcznym rozwiązaniu konfliktu:

  • usuń wszystkie znaczniki konfliktu,
  • dodaj pliki do Git (git add .) i kontynuuj merge (git merge --continue)
